# CLAUDE.md ジェネレーター

リポジトリを分析し、Claude Code向けの最適なCLAUDE.mdを生成・更新するスキル。

## 設計思想

CLAUDE.mdは「リポジトリの北極星」。Claude Codeがシニアエンジニアとして振る舞うには、
以下の4つを常に把握させる必要がある：

- **Why** — このシステムは何をするのか
- **Map** — どこに何があるのか
- **Rules** — 何をしていい/ダメなのか
- **Workflows** — どうやって作業を進めるのか

CLAUDE.mdは**短く保つ**ことが最重要。長すぎるとコンテキストが抜け落ちる。
詳細は別ファイルに委譲し、CLAUDE.mdからポインタを張る構成にする。

## 実行手順

### Step 1: リポジトリ分析

以下を並列で調査する：

1. **技術スタック検出**
   - `package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `Gemfile` 等を探す
   - フレームワーク・言語・ビルドツールを特定

2. **ディレクトリ構造把握**
   - トップレベルのディレクトリ一覧を取得
   - `src/`, `app/`, `lib/`, `packages/` 等の主要ディレクトリの役割を推定

3. **既存設定ファイル確認**
   - 既存の `CLAUDE.md`（ルート、`.claude/`、サブディレクトリ）
   - `.claude/settings.json`（hooks設定）
   - `.claude/skills/`（既存スキル）
   - `docs/` ディレクトリの有無と内容
   - `.editorconfig`, `.prettierrc`, `eslint.config.*` 等のコーディング規約

4. **Git情報**
   - 最近のコミットメッセージのスタイル（Conventional Commits等）
   - ブランチ戦略（main/develop等）
   - `.gitignore` の内容

5. **リスクゾーン特定**
   - `auth/`, `api/`, `middleware/`, `config/`, `infra/`, `db/migration/` 等
   - セキュリティ上注意が必要なディレクトリを列挙

### Step 2: CLAUDE.md 生成

分析結果をもとに以下の構造でCLAUDE.mdを生成する。

#### テンプレート構造

```markdown
# [プロジェクト名]

[1-2文でプロジェクトの目的を説明]

## 技術スタック

[言語、フレームワーク、主要ライブラリをリスト]

## ディレクトリ構造

[主要ディレクトリと役割を簡潔に記載。全ファイルを列挙しない]

## 開発コマンド

[ビルド、テスト、lint、起動等の主要コマンド]

## コーディング規約

[プロジェクト固有のルール。汎用的すぎるものは書かない]

## ワークフロー

[ブランチ戦略、コミットメッセージ規約、PR作成ルール等]

## 注意事項

[やってはいけないこと、特に気をつけるべきこと]
```

#### 生成ルール

- **50行以内**を目標にする。超える場合は詳細を別ファイルに委譲
- package.jsonのscriptsやMakefile等から開発コマンドを正確に抽出する
- 既存のCLAUDE.mdがある場合は内容を尊重しつつ不足を補完する
- `@ファイル名` で別ファイルを参照できることを活用する
- 推測で書かない。分析で確認できた事実のみ記載する

### Step 3: モジュール別ローカルCLAUDE.md の提案

リスクゾーンとして特定したディレクトリに対し、ローカルCLAUDE.mdの配置を提案する。

#### 提案基準

| ディレクトリパターン | 提案する理由 |
|-------------------|------------|
| `auth/`, `authentication/` | 認証ロジックの変更は影響範囲が広い |
| `api/`, `routes/` | APIの変更は後方互換性に注意が必要 |
| `db/`, `migration/`, `prisma/` | データベーススキーマ変更は不可逆になりうる |
| `infra/`, `terraform/`, `k8s/` | インフラ変更は本番環境に影響する |
| `config/` | 設定変更は全体に波及する |

#### 提案フォーマット

各候補について以下を提示する：
- 配置パス
- 記載すべき内容の概要（3-5行）
- 作成するかどうかをユーザーに確認

ユーザーが承認したもののみ作成する。

### Step 4: hooks 設定の提案

`.claude/settings.json` の hooks 設定を分析し、改善を提案する。

#### 提案例

- **フォーマッター**: Edit後に `prettier --write` や `rustfmt` を実行
- **テスト**: テストファイル変更時に関連テストを実行
- **lint**: コミット前にlintを実行
- **セキュリティ**: `auth/`, `config/` 等への書き込み時に警告

既存のhooksがある場合は重複しないよう注意する。
提案のみ行い、ユーザー承認後に適用する。

### Step 5: docs/ 構成の提案

`docs/` ディレクトリの構成を分析し、段階的コンテキストとしての活用を提案する。

#### 推奨構成

```
docs/
├── architecture.md    # アーキテクチャ概要
├── adr/               # 設計判断記録（ADR）
└── runbooks/          # 運用手順書
```

既存の `docs/` がある場合はその構成を尊重する。
CLAUDE.mdから `@docs/architecture.md` 等で参照する形を推奨する。

### Step 6: ユーザーへの提示とフィードバック

生成した内容をすべてユーザーに提示し、以下を確認する：

1. ルートCLAUDE.mdの内容
2. モジュール別CLAUDE.mdの配置（承認されたもののみ作成）
3. hooks設定の変更（承認されたもののみ適用）
4. docs/構成の変更（承認されたもののみ作成）

**ユーザーの承認なしにファイルを作成・変更しない。**

## 更新モード

既存のCLAUDE.mdがある場合の更新ルール：

- ユーザーが手書きした内容は削除しない
- 新しく検出された情報を追記する形で更新
- 構造が大きく変わる場合はbefore/afterを提示して確認を取る
- `@ファイル名` の参照先が存在するか検証する

## アンチパターン（やらないこと）

- CLAUDE.mdを100行以上に膨らませない
- 「一般的なベストプラクティス」を書かない（プロジェクト固有の情報のみ）
- 推測でコマンドやパスを書かない
- ユーザーの既存設定を勝手に上書きしない
- `.env` やシークレットに関する情報をCLAUDE.mdに含めない
